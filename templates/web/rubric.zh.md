# 评估 Rubric — Web 前端

按当前 active plan 的需求（不是代码自称完成了什么）给每个维度打 **1–5** 分。**任一维度 < 3 = FAIL。** 使用 `.harness/config.json` 里的 `verificationRecipe` 收集证据——实际运行测试/工具并阅读完整输出。

## 1. 功能性（硬门）
- 变更行为对应的 e2e/unit 测试通过（`npx playwright test` / 项目测试入口）。
- 功能按计划要求工作，包括空状态、加载状态和错误状态。
- 5 = 所有状态都有通过的测试覆盖；3 = 快乐路径有覆盖；1 = 测试失败或没有证据。

## 2. 视觉 / 可访问性
- Playwright 截图 / 视觉回归：相对 baseline 没有非预期布局退化。
- 键盘焦点顺序和基本 a11y（label、role、contrast）保持正确。
- 5 = 符合规格且 a11y 已验证；3 = 可接受但有轻微打磨债；1 = 布局或 a11y 明显退化。

## 3. 集成
- 既有流程的 e2e 套件仍通过（无回归）。
- API/契约边界被遵守；服务端关注点没有泄漏进 UI。
- 5 = 全套绿色且边界清晰；3 = 无明显回归；1 = 破坏既有流程。

## 4. 代码质量
- `harness-verify` 退出码为 0（eslint + tsc + layering）。
- 没有用 `any` 掩盖真实类型错误，没有 console/debug 遗留，没有死代码。
- 5 = 干净、类型充分、符合项目惯用法；3 = 可工作但有债；1 = lint/type 错误或分层违规。

## 结论
- PASS = 每个维度 ≥ 3。WARN = 全部 ≥ 3 且有维度 == 3（标记技术债）。FAIL = 任一维度 < 3（列出修复项）。
